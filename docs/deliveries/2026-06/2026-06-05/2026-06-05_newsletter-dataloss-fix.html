The Scare

On the deployed app the Marketing block text + link showed as placeholders ([Button link]) and the photos were gone. Git was clean and pushed, Vercel matched — not a deploy problem. A direct database query showed the saved record nl_may_2026_roundup was fully intact. The editor was showing a stale draft restored from the browser's local storage (sprout_nl_editor_v1), from before the marketing copy + photos were added. Clearing it by hand failed because the running app rewrites the snapshot on every keystroke.

Fix 1 — Necessary

stops the recurrence

The restore effect loaded the local snapshot verbatim and never checked the database. It now prefers the authoritative DB record for any saved draft: it waits for the newsletters list to load, then matches by id. Only a never-saved new draft falls back to the cache.

A stale cache can no longer override saved content.

630cf7b

Fix 2 — Hardening

no silent failures

New single-record saveNewsletter(n) that validates and surfaces a real error on failure, instead of silently skipping and reporting a false "Saved ✓". Writes only the changed record via a functional state update; strips _isNew/_wasNew so editor flags never persist.

cd30eba

Blast Radius & Decisions

  • Newsletter-only. Contacts, orgs, events, and profile use separate save functions and are untouched. saveNewsletter is referenced only by services.js + CRMManager.jsx.
  • Two residual items deliberately NOT added (they would only guard cases Fixes 1+2 already prevent → bloat): the best-effort beforeunload tab-close save, and a post-save refetch.
  • Data never lost. The DB was verified intact before and after the browser was closed; the async close-save is cancelled by the browser, which is why the record survived.
