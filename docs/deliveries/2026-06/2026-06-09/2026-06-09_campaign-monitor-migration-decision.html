Planning / Decision No code change 2026-06-09

Migrate the newsletter email channel:
Givebutter → Campaign Monitor

Build the newsletter in the CRM, send its full custom HTML natively via Campaign Monitor's API.
Why switch

Full HTML import

Givebutter Engage is template-only — it rejects raw HTML, so the custom Sprout template (crop boxes, colored blocks, footer buttons) can't be used. Campaign Monitor accepts full HTML, so buildCompact() renders exactly as designed.

Why now

Scale: ~3,750 contacts

The 06-09 import pushed the CRM past 3,700 contacts. The current Gmail-BCC send (45/batch, 2,000/day cap, no real unsubscribe) won't scale. A real ESP was needed regardless of the HTML issue.

What changes
Gmail BCC send → Campaign Monitor API · Givebutter = donations (unchanged)

The in-app Send card flow (passphrase → test → approve → send to list) stays identical. Only the engine underneath swaps.

Decisions locked

Settled this session

  • Full API send from inside the CRM
  • Deliverability concern dropped — most contacts already donated/attended
  • The 20 "Campaign Monitor" CSVs came from the existing account → already-consented
To build (when unblocked)

Two pieces

  • Contact sync — CRM → CM lists by bucket (Community / Donors / Prospects), matched on email
  • Rewrite app/api/send — create a campaign from the HTML, trigger send via CM's API
  • Point footer unsubscribe at CM's merge tag
