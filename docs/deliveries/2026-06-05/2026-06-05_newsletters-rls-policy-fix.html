Newsletter saves fixed — added the missing RLS policy to sprout_newsletters

“Save failed: new row violates row-level security policy” → resolved at the database, no app code, no redeploy.
🛡️ Supabase MCP 🗄️ DB-only change ⚡ Effort: diagnose low / fix low 📅 2026-06-05
What was wrong

🔍 Root cause

sprout_newsletters had RLS enabled but zero policies. Postgres denies every write by default — so the table was effectively locked and newsletters had never persisted through the app.

It was the only sprout_* table missing a policy. Every other CRM table already carried an identical anon_all rule.

🧭 How we confirmed

Queried pg_policies across all sprout_* tables — newsletters returned no rows while contacts / orgs / events / profile each had anon_all (using true / with check true).

The fix

✅ Applied via Supabase MCP

Added the same anon_all policy through apply_migration — migration name sprout_newsletters_anon_all_policy. Verified live in pg_policies; matches the other tables exactly. Applies immediately, no redeploy.

create policy "anon_all" on public.sprout_newsletters for all to anon using (true) with check (true);

🔐 Security reality (deferred)

This app has no Supabase Auth — RLS is on everywhere but every policy is wide-open anon, and the anon key ships in the client bundle. The whole CRM is effectively public.

So this change makes newsletters consistent, not less safe. Real lockdown (Auth + auth.uid()-scoped policies on every table) is now tracked as its own cross-app project.
