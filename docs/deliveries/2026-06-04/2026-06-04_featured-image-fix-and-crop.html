Fixed The root cause: HTML leaking into the alt attribute

The featured (and spotlight) <img alt="…"> used cval() for the alt text — which returns an HTML <span> placeholder when the field is empty. Those quotes terminated the <img> tag early, so the browser dropped the real style (height:200px; max-width:544px; object-fit:cover) and the photo fell back to natural size = gigantic.

Why it hid for 4 sessions: every prior fix tweaked the sizing CSS, and every test used a filled-in title (so cval returned clean text). Uploading the photo before typing a title triggered the empty-field path. The smoking gun: inspecting the live element showed alt="<span style=" color:#c2c2bf;"…

Fix: both alts now use plain escaped text — esc(v.featuredTitle || "Featured event"). Verified in headless Chromium with an empty title → renders exactly 544×200.

New Drag-to-reposition crop for the featured photo

Since the image already uses object-fit:cover, repositioning just controls object-position — no canvas, no re-upload, email-safe. New reusable ImageCrop component shows the photo at the exact 544×200 banner shape; drag the photo to choose what shows. Saves as field_values.featuredPhotoPos ("50% 35%"), recenters on Replace.

Needs OK sprout_newsletters has no RLS policy

It's the only sprout_* table with no row-level-security policy, so every save returns 401 and the table is empty — drafts & crop positions don't persist, and the editor floods with "Save failed". One-line fix ready (the same anon_all policy every other table has). Say "add the newsletters policy" to apply.
